00lifecycle
From blank context to shipped code.
three commands · six phases · one source of truth
01what it does

Three jobs.
One install.

Sprint-workflow doesn't replace Claude Code — it directs it. The plugin enforces a structured lifecycle so Claude doesn't write whatever it feels like at the moment, and so the work it produces is traceable, reviewable, and tied to a real plan.

A  /  ARCHITECT

Turn a conversation into a roadmap.

/sprint-architect takes any context (the current session, a PRD, attached docs) and produces a Linear Project, an Architecture & Roadmap document (C4 + ADR + arc42), and Epic issues for each phase. Asks ≤5 clarifying questions, each with a "Choose for me" option.

B  /  ORCHESTRATE

The sprint lead never writes code.

It dispatches. Backend, frontend, tests, security, DBA, docs, and product-management agents run in parallel where they can, sequentially where they must. Codex handles surgical execution; Claude reviews. Status moves through Linear, comments are attributed, the user can step in.

C  /  VERIFY

Build, lint, test, review — then ship.

Phase 3 runs build + lint + tests + spec compliance + a Codex adversarial review + a Claude code review — in parallel. Architecture drift is caught and classified: drift = warning, erosion of an Accepted ADR = blocking. Fix loop runs until clean.

02orchestration

The sprint lead orchestrates —
nothing more, nothing less.

The orchestrator never writes a line of code. Its only job is to dispatch the right agent with the right context, track results, retry failures, and finalize tracking. That separation is what keeps work traceable and quality high.

fig. 02 · agent network orchestrator + 9 specialists, by responsibility
↔ swipe to see the full network
Why this matters. Most "AI agent" setups conflate planning with execution. Here they're separate: the orchestrator picks the work, agents do the work, gates validate the work. When something fails, the orchestrator routes the fix — surgical fixes go to Codex, architectural fixes go back to the original agent. No task ships with known violations. — from the plugin's CLAUDE.md
03linear is the source of truth

The architecture lives where
the team already looks.

Local docs/architecture.md goes stale within weeks because nobody opens it during day-to-day work. Sprint-workflow puts the architecture & roadmap in a Linear Project Document — on the page where Epics and Tasks already live — and links every level back to it.

fig. 03 · project hierarchy linear · everything resolves up to the document
↔ swipe to see the full hierarchy
Drift vs erosion. Drift = code introduces a component or edge the architecture doc doesn't mention. Often correct, just undocumented — a warning. Erosion = code violates an Accepted ADR or quality attribute — blocking. The check runs in /sprint-plan, /sprint-enrich, Phase 3 QA, code review, and /sprint-retro. When erosion is intentional, run /sprint-architect --update — the new decision becomes an ADR, the old one is superseded, the next sprint passes. — architecture-drift-check skill
04codex delegation · opt-in

Conserve tokens.
Get a second pair of eyes.

When the codex-plugin-cc is installed, sprint-workflow auto-detects it and routes scoped execution to OpenAI Codex (~4× more token-efficient for well-bounded work) while Claude handles planning, architectural decisions, and code review. Cross-model verification — Codex catches issues Claude-reviewing-Claude would rationalize away.

phase 1 · task

Task assignment

⚡ codex /codex:rescue — tasks flagged codex-eligible at planning time
claude Original opus agent — everything else
phase 3 · qa

QA gate

⚡ codex /codex:adversarial-review — replaces the qa-agent entirely when opted in
claude qa-agent (sonnet) — fallback when Codex unavailable
phase 3 · review

Code review

always pr-review-toolkit:code-reviewer on Claude — never Codex. Claude's idiomatic pattern recognition is superior for review.
phase 4 · fix

Fix routing

⚡ codex Surgical fixes — single file, mechanical (lint, null check, missing import, typo)
claude Architectural fixes — multi-file, interface change, design decision needed
✓ Codex-eligible
  • Well-scoped CRUD — "Create REST endpoint for listing orders with pagination"
  • Migration scaffolds — "Add migration for new audit_logs table"
  • Boilerplate generation — controllers + services + repositories
  • Test generation — against existing implementation, straightforward assertions
  • Single-concern features — clear acceptance criteria, no ambiguity, no local services
✗ Not Codex-eligible
  • Cross-cutting architecture — "Redesign middleware pipeline for multi-tenancy"
  • Design exploration — multiple approaches to compare
  • Local services needed — running PostgreSQL with seed data, Docker, VPN
  • Security-critical — OAuth2 PKCE flows, token rotation, audit-trail design
  • Heavy iteration expected — UX exploration, complex state management
Auto-detection · no flags needed. Sprint-workflow checks for /codex:rescue + /codex:adversarial-review at the start of every sprint command. If both are present, Codex mode activates and the product-manager flags eligible tasks during planning. If either is missing, every task routes to Claude. Same orchestration either way — Codex is purely additive.
05visibility & control

Watch agents work.
Reject, comment, move things around.

Every dispatch, every status change, every fix lands as a comment on the relevant Linear issue — attributed to the agent that did it. You can reply, drag a card back across the board, or override anything at any time. The sprint lead picks up your changes on the next turn. No more grepping a stale SPRINT_PLAN.md to figure out where things are.

01 · transparent

Agents narrate as they work

Each phase transition and result lands as a comment on the issue, attributed to the agent. You see why a task moved, not just that it moved. HANDOFF blocks from worktree-isolated agents land here too — branch, SHA, files, build status.

02 · bidirectional

You can override at any moment

Drag a card from In Review back to In Progress with a comment explaining why. The orchestrator detects the manual transition on its next pass and re-dispatches the agent with your note as additional context. Same for cancelling, re-prioritising, or changing assignment.

03 · always current

Source of truth stays alive

Agents update Linear as they work, so the board reflects reality. There's no parallel markdown going stale because nobody opens it. Every Task resolves up to its Epic, every Epic up to its Project Document — queryable in one place.

06status lifecycle

07nine personas

Specialist agents,
each with their own conventions.

Every agent reads its required skill files before writing a line of code. Project-local skills override plugin-bundled ones. When a task references a Linear Epic, agents fetch the parent Project's Architecture & Roadmap automatically.

B
backend-dev codex-elig
opus · server-side

Server-side work across any stack — .NET, Node, Python, Go, Rust. API endpoints, service logic, middleware, infrastructure.

F
frontend-dev codex-elig
opus · client UI

Client-side UI — React, Vue, Svelte, Angular. Components, pages, styling, state, API integration.

D
dba-agent
opus · data layer

Schema design, migrations, indexing, PII/POPIA compliance audits, expand-contract patterns for zero-downtime change.

S
security-agent
opus · audit + harden

OWASP Top 10 audits, auth/authz patterns, secret + PII scanning, dependency vetting, audit trails.

T
test-writer codex-elig
sonnet · tests

Unit, integration, snapshot tests across any framework. Detects test framework, follows existing patterns. Regression-test-first for bug fixes.

Q
qa-agent codex-replaces
sonnet · quality gate

Build, lint, type-check, test runs. Spec-compliance audits. Architecture drift checks during Phase 3. When Codex is opted in, replaced entirely by /codex:adversarial-review.

P
qa-playwright
sonnet · e2e + a11y

E2E browser automation, visual regression baselines, accessibility (axe-core), Page Object Model.

W
docs-agent
sonnet · documentation

Technical docs, READMEs, ADRs (MADR format), changelogs from conventional commits, OpenAPI from code.

M
product-manager
opus · planning + architecture

Sprint planning, INVEST validation, MoSCoW prioritization, Given/When/Then acceptance criteria, architecture mode for /sprint-architect.

09/sprint-architect

From conversation
to a living architecture doc.

Run /sprint-architect with no arguments and the main session uses the current conversation as context. Confirm what it understood, answer ≤5 clarifying questions (each with a "Choose for me" option), and out comes a Linear Project Document, ready to drive every sprint that follows.

linear.app · order-rewrite · documents · architecture & roadmap
PROJECT DOCUMENT · v1

Architecture & Roadmap

Migrate the legacy monolith order pipeline to a bounded-context architecture, phased over four sprints. Initial users: warehouse + customer support.

3 · containers (c4 level 2)
Orders API · .NET 8 · sync HTTP from Frontend · events to EventBus, SQL to OrdersDB
Inventory · .NET 8 · events from Orders · SQL to InventoryDB
Catalog · .NET 8 · sync HTTP from Frontend · SQL to CatalogDB (read-only replica)
6 · architectural decisions
ADR-3 · Status: Accepted auto-selected
Decision: Cross-context communication is async via EventBus, not sync HTTP.
Consequences: Forbids direct HTTP calls between Orders ↔ Inventory ↔ Catalog. Drift check will flag any sync edge.
ADR-4 · Status: Accepted
Decision: No shared databases across bounded contexts. Each service owns its schema.
Consequences: Forbids cross-context FKs. Reads happen via async snapshots or HTTP read endpoints.
7 · change log
Empty — populated by /sprint-architect --update when reality shifts.
"Choose for me" UX. For every clarifying question, you get 2–4 concrete options plus Choose for me and Skip / not sure. Auto-selected decisions are tagged in the doc so you can review and override them later via /sprint-architect --update. The cap is 5 questions per run — further decisions get safe defaults the user can ratify in their own time.
10install

Up and running
in under a minute.

Sprint-workflow is a Claude Code plugin. Install via the marketplace, then opt into Linear (for sprint tracking) and Codex (for token-efficient task execution) when you're ready.

Add the marketplace

In Claude Code, register this repo as a plugin marketplace.

$ /plugins marketplace add rynhardt-potgieter/sprint_workflow

Install the plugin

One install brings 9 agents, 23 skills, 13 commands, and the hooks.

$ /plugins install sprint-workflow

Optional: Linear MCPrequired for /sprint-architect

Auto-detected at runtime. Without it, sprint tracking falls back to a markdown file.

# Follow Linear's MCP setup
$ open https://linear.app/docs/mcp

Optional: Codex CLI~4× token savings

Auto-detected. When available, scoped tasks route to Codex; reviews stay on Claude.

# Install codex-plugin-cc to wire it in
$ /plugins install codex

Start an initiative

Already had a brainstorm in this conversation? Use the current chat as context.

$ /sprint-architect
# Confirm context, answer up to 5 questions, get a Linear Project + Doc + Epics

$ /sprint-plan PROJ-122
# Auto-loads Architecture & Roadmap, breaks the Epic into Tasks

$ /sprint-start
# Phases 1-6: implement, test, QA + review, fix, docs, ship
11inside the plugin

Everything's just files.
Read them, fork them, override them.

Skills are markdown. Agents are markdown with a frontmatter prompt. Commands are markdown with a YAML header. Hooks are bash scripts. There's no DSL, no compiler, no build step — the contract is whatever Claude Code reads.
