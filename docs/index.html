01 · what it actually does

Claude already dispatches subagents.
This gives you somewhere to watch them.

Claude Code can already plan, delegate to subagents, and break work into steps. It just does it all in the chat context, in temp files, or in a sprint plan markdown — surfaces nobody on your team actually opens. You re-prompt; you don't review. sprint-workflow doesn't add delegation. It adds a second surface — Linear — where every dispatch, every status change, every agent comment becomes something you can see, comment on, reject, or override.

Without sprint-workflow

Subagents in the dark

Claude does delegate. The plan, the dispatch, the sub-results — all of it lives in the chat context, in temp files, or in a SPRINT_PLAN.md nobody opens after week one.

  • Planning happens. You can't see it as it happens.
  • Subagents run. You can't comment on what they're doing.
  • If you want to redirect, your only verb is "re-prompt"
  • Architectural decisions vanish into chat history
  • To audit, you re-read the transcript
With sprint-workflow

The same agents, on a surface you can grab

Same planning, same delegation. But every step lands in Linear — the same board your team already uses for tickets. You participate; you don't just spectate.

  • Each dispatch posts a comment on the issue, attributed to the agent
  • You drag a card back across the board — orchestrator picks it up next turn
  • Decisions live as ADRs in the Project Document
  • Drift between code and architecture gets flagged before merge
  • To audit, you read a board. To intervene, you leave a comment.
02 · the team

Nine specialists, one sprint lead.

04 · the lifecycle

Three commands.
Six phases. One source of truth.

You drive the workflow with three commands. Each one expands into a defined sequence of steps, dispatches, and gates. The sprint lead handles the rest.

STAGE 01

Architect

Turn any context (this conversation, a PRD, attached docs) into a Linear Project, Architecture & Roadmap document, and Epic issues for each phase. Asks ≤5 clarifying questions.

/sprint-architect
STAGE 02

Plan & Enrich

Pick an Epic. Break it into tasks with agent assignments and acceptance criteria. Specialist agents enrich with security, DB, and test concerns.

/sprint-plan + enrich
STAGE 03

Execute & Ship

The sprint lead runs the six phases below: build → test → QA + review → fix → docs → ship. Failures loop back. Linear stays current.

Codex

And the architecture document stays honest.

Every time the implementation diverges from what the architecture document says, sprint-workflow flags it — at planning time, at QA time, at code-review time, and in the sprint retro. There are two things we look for:

DRIFT · warning

Code adds something the doc doesn't mention

Often correct — the doc was incomplete and reality has more nuance. Flagged so you can decide whether to update the doc.

Example: ADR-3 says "Orders → Inventory via EventBus." Code adds a Redis cache between them. The doc is silent on caching. Warning, not blocking.
EROSION · blocking

Code violates an explicit architectural decision

The doc said don't, the code did. The sprint can't ship until you either revise the code or update the architecture document with a new ADR.

Example: ADR-3 forbids sync HTTP between bounded contexts. Code adds httpClient.GetAsync(inventoryUrl) from Orders. Blocking.
06 · codex routing · optional

Cheaper execution.
Cross-model verification.

Codex is OpenAI's command-line coding agent. When you have it installed, sprint-workflow auto-detects it and routes scoped tasks to Codex instead of Claude — roughly four times more token-efficient for well-bounded work, and the model swap means quality reviews happen across two model families instead of one.

How the routing decides Opus Codex

Every agent runs on Opus by default. If Codex is installed AND the task is flagged eligible (clear scope, single concern, no design ambiguity), the work routes to Codex automatically. If Codex isn't installed, every task runs on Opus and the page works exactly the same. If a Codex run fails, sprint-workflow falls back to Opus on its own. You don't pick — the orchestrator does, and you can see which one ran on every issue.

✓ Routes to Codex
  • Well-scoped CRUD — "REST endpoint for listing orders with pagination"
  • Migration scaffolds — "Add migration for new audit_logs table"
  • Boilerplate — controllers + services + repositories
  • Tests against existing code — straightforward assertions, clear setup
  • Single-concern features — no ambiguity, no local services needed
✗ Stays on Opus
  • Cross-cutting architecture — "Redesign middleware for multi-tenancy"
  • Design exploration — multiple approaches to compare
  • Local services needed — running Postgres, Docker, VPN
  • Security-critical — OAuth flows, audit trails
  • Heavy iteration — UX exploration, complex state

08 · what's required, what's optional

One thing required.
Two powerful add-ons.

The plugin works on its own. Linear and Codex are opt-in — install them when you're ready and sprint-workflow auto-detects them. You don't change any config; the plugin notices and switches behaviour.

Required

Claude Code

The host environment. sprint-workflow installs as a Claude Code plugin, alongside whatever else you already use. Free.

Required
Optional

Linear MCP — project tracking

Without it: sprint plans live in a markdown file in your repo. Functional, but goes stale. With it: plans, the Architecture & Roadmap document, and every status change live in Linear — required for /sprint-architect.

Optional required for /sprint-architect
Optional

Codex CLI — cheaper execution

Without it: every task runs on Claude Opus. With it: eligible tasks route to Codex (~4× more token-efficient for scoped work), and Phase 3 QA gets a cross-model adversarial review. Falls back to Opus automatically if Codex isn't available or fails.

Optional auto-detected at runtime
09 · install

Up and running
in under a minute.

Two commands to install. Two more to wire up the optional integrations when you're ready.

01

Add the marketplace Required

In Claude Code, register this repo as a plugin marketplace.

$ /plugins marketplace add rynhardt-potgieter/sprint_workflow
02

Install the plugin Required

One install brings 9 agents, 23 skills, 13 commands, and the hooks.

$ /plugins install sprint-workflow
03

Linear MCP Optional

Required for /sprint-architect. Without it, plans live in a markdown file.

# Follow Linear's MCP setup
$ open https://linear.app/docs/mcp
04

Codex CLI Optional

Auto-detected. When available, eligible tasks route to Codex automatically.

# Install codex-plugin-cc to wire it in
$ /plugins install codex
05

Start an initiative

Already had a brainstorm in this conversation? Use the current chat as context.

$ /sprint-architect
# Confirm context, answer up to 5 questions, get a Linear Project + Doc + Epics

$ /sprint-plan PROJ-122
# Auto-loads Architecture & Roadmap, breaks the Epic into Tasks

$ /sprint-start
# Phases 1–6 — implement, test, QA + review, fix, docs, ship
