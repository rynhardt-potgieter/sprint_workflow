## Tracking Mode Detection

### Linear MCP Check
1. Look for available MCP tools matching `mcp__linear__*` or `mcp__claude_ai_Linear__*`
2. Try calling `list_teams` with whichever prefix exists
3. If it succeeds → **Linear mode**. Read `${CLAUDE_PLUGIN_ROOT}/skills/linear-sprint-planning/SKILL.md`.

5. If no Linear MCP tools exist → **MD mode** (default)

## Your Task

Produce a single markdown handoff file that any future session — yours or another agent's — can read in under 30 seconds and use to resume the sprint exactly where it stopped.

This command **never** writes code. It only reads state and writes the handoff document.

### 1. Determine Output Path

If `$ARGUMENTS` provides a path, use it. Otherwise default to `docs/SPRINT_HANDOFF.md`.

If a handoff file already exists at the target path, **read it first**, then overwrite with the fresh snapshot. The previous snapshot's "Next Action" is useful context for the new one.

### 2. Gather Sprint State

**Linear mode:**
1. Discover team/project (ask user to confirm if ambiguous, or reuse from session context)
2. Find the active sprint Milestone — `list_milestones`, pick the one with the most `In Progress` issues
3. Query Stories: `list_issues` filtered by `milestoneId` and `Epic` label
4. For each Story, query Tasks: `list_issues` with `parentId`
5. Capture for every issue: `id`, `title`, `status`, last comment timestamp, `Agent`/`Phase` fields parsed from description
6. Note Stories with **all** Tasks in `Done` (sprint-complete by Story)
7. Note Stories where ANY Task is `In Progress` (this is where we resume)

**MD mode:**
1. Locate the active plan: `docs/SPRINT_PLAN.md`, `docs/SPRINT*.md`, `SPRINT_PLAN.md` (in that order)
2. Parse stories, tasks, and their statuses (`not-started`, `in-progress`, `implementation-complete`, `tests-written`, `review-complete`, `fixes-applied`, `documented`, `committed`, `completed`)
3. Capture last-modified timestamp of the plan file

### 3. Determine Current Phase

Map task statuses to the 6-phase model in `sprint-start.md`:

| Distribution of statuses | Current Phase |
|---|---|
| Any task `In Progress` (Linear) or `in-progress` (MD) | Phase 1 — Implementation |
| All implementation tasks `In Review` / `implementation-complete`, no test artefacts yet | Phase 2 — Tests |
| Tests present, no QA comment / status update | Phase 3 — Quality Gates |
| QA returned BLOCKING and tasks bounced back to `In Progress` | Phase 4 — Fix Loop |
| All tasks `In Review` / `review-complete`, no docs commit | Phase 5 — Documentation |
| All tasks `In Review` / `documented`, no commit/checkin | Phase 6 — Commit & Push |
| All tasks `Done` / `completed` | Sprint Complete |

If the distribution is ambiguous (e.g., some tasks `In Progress` AND some in `In Review`), report **Phase 1 (mixed)** and list both groups separately.

### 4. Identify Blockers

Scan for blockers in:

- **Linear**: comments mentioning "blocked", "waiting for", or status `Blocked` if used. Issues open >7 days with no status change.
- **MD**: lines flagged `BLOCKED:`, `WAITING:`, or stories not progressing across 2+ status updates.
- **Git**: uncommitted changes (`git status --porcelain`) that suggest in-flight work; unmerged conflicts; rebases/cherry-picks in progress (`.git/rebase-*`).
- **Build state**: not run by this command (it's read-only) — but note if the last QA report on file flagged blocking issues.

### 5. Write the Handoff Document

Use this exact template — keep it under 200 lines:

```markdown
# Sprint Handoff — <project> — <sprint name> — <YYYY-MM-DD HH:MM>

> Generated by `/sprint-handoff`. A future session reads this to resume.
> Source of truth: <Linear milestone <name> | docs/SPRINT_PLAN.md>

## Current Phase

**<Phase N — Name>**

<one paragraph: what was last done, what is in flight>

## In-Flight Tasks

| Task | Story | Agent | Status | Last Update |
|------|-------|-------|--------|-------------|

(Include only tasks NOT in `Done` / `completed`. Sort by Story, then by status.)

## Last Successful Phase

<Phase N — Name> — completed at <timestamp or "unknown">

<one line: e.g., "Phase 2 tests written for Stories 1–3, all passing locally">

## Blockers

- <Blocker 1: what, where, who/what is needed to unblock>
- <Blocker 2: ...>

(If none: "None identified.")

## Uncommitted / Unresolved Local State

- Uncommitted files: <count> (<git status --short>)
- Rebase / cherry-pick in progress: <yes/no>
- Last failing build/test: <yes/no, link to log if available>

## Resume Command

```
/sprint-continue
```

(Or `/sprint-resume-task <id>` if only one task is in flight.)

## Notes for the Next Session

- <Anything non-obvious. E.g., "Codex was used for tasks 3 and 5; do not re-route them to Claude.">
- <E.g., "Linear MCP returned -32600 twice during Phase 1; retry succeeded — flag to user if it happens again.">
- <E.g., "Story 2 acceptance criterion #4 is ambiguous — see comment on T-123.">
```

### 6. Confirm and Exit

After writing:

```
✓ Handoff written to <path>

Phase: <Phase N>
In-flight tasks: <count>
Blockers: <count>

To resume: /sprint-continue
```

Do NOT dispatch any agents. Do NOT update Linear / MD plan status. This command is **read-only** with respect to the sprint — it only produces the handoff document.

---

## When To Run This Command

- Before walking away from a partially complete sprint
- When usage limits are about to hit
- Before switching machines
- After a network/MCP outage where you want to capture state before retrying
- At end of day on any active sprint

Re-running it overwrites the previous handoff. That's fine — the freshest snapshot is the only one that matters.
